# BlogTool 桌面应用发布操作手册

本文档只说明 **Windows 安装包发布**，不涉及博客内容发布。

当前项目支持两种发布方式：

1. **本地发布**：在你自己的电脑上执行打包，必要时直接把安装包发布到 GitHub Releases
2. **自动发布**：代码推到 GitHub 后，由 GitHub Actions 自动打包并发布到 GitHub Releases

## 一、发布产物与更新机制

### 1. 最终安装包

- 安装包文件名：`BlogTool-Setup-<version>.exe`
- 默认输出目录：项目根 `release/`

### 2. 自动更新依赖

已安装应用的自动更新依赖 **GitHub Releases**：

- 新版本必须发布到 GitHub Releases
- 安装版应用启动后会自动检查 GitHub Releases
- 发现新版本会后台下载
- 下载完成后提示安装，或在下次退出时自动安装

因此，**只有打包但不发布到 GitHub Releases，不足以让已安装客户端收到更新。**

## 二、本地发布

本地发布适合以下场景：

- 先在本机验证打包是否成功
- 需要手工控制什么时候发布到 GitHub Releases
- 临时抢修一个版本，不想先走 GitHub Actions

### 1. 前置条件

执行本地发布前，确保满足以下条件：

- 已安装 Node.js 22
- 已执行过 `npm install`
- 当前工作区代码已经是你要发布的版本
- 若要**直接发布到 GitHub Releases**，还必须准备：
  - `GH_TOKEN`
  - `GITHUB_REPOSITORY=owner/repo`

`GH_TOKEN` 需要具备当前仓库的发布权限。

### 2. 先只在本地打包，不发布到 GitHub

执行：

```powershell
& 'F:\Program Files\nodejs\npm.cmd' run dist:win
```

等价逻辑：

- 先执行前端构建 `npm run build`
- 再执行 `electron-builder --config electron-builder.config.cjs --win nsis --publish never`

成功后，产物在：

- `release/BlogTool-Setup-<version>.exe`
- `release/*.blockmap`

这种方式**不会**上传到 GitHub Releases。

### 3. 在本地直接发布到 GitHub Releases

先设置环境变量：

```powershell
$env:GH_TOKEN = '你的 GitHub Token'
$env:GITHUB_REPOSITORY = 'owner/repo'
```

再执行：

```powershell
& 'F:\Program Files\nodejs\npm.cmd' run dist:win:publish
```

等价逻辑：

- 先执行前端构建 `npm run build`
- 再执行 `electron-builder --config electron-builder.config.cjs --win nsis --publish always`

成功后会发生两件事：

1. 本地生成安装包
2. 直接把安装包和更新元数据上传到 GitHub Releases

### 4. 本地发布的推荐顺序

推荐你按这个顺序操作：

1. 修改 `package.json` 里的 `version`
2. 本地执行 `npm run dist:win`
3. 安装并验证应用启动、托盘、自动更新、数据库路径等行为
4. 确认没问题后，再执行 `npm run dist:win:publish`

这样可以避免把未经验证的安装包直接推到 Releases。

### 5. 本地发布后的检查项

发布完成后，至少检查：

- GitHub Releases 中是否出现对应版本
- Release 资产中是否包含 `BlogTool-Setup-<version>.exe`
- 是否存在更新元数据文件（通常是 `latest.yml` 或同类 `.yml` 文件）
- 已安装旧版本客户端是否能检测到更新

## 三、提交后自动发布

当前自动发布不是“普通 commit 一推就发布”，而是：

- **推送版本标签 `vX.Y.Z` 时自动发布**

也就是说：

- 只 `git push` 分支：**不会发版**
- 必须推标签：**才会触发正式 Release 构建**

对应工作流文件：

- [.github/workflows/release-desktop.yml](../.github/workflows/release-desktop.yml)

### 1. GitHub 仓库一次性准备

首次使用自动发布前，确认以下条件：

- 代码仓库已经在 GitHub 上
- Actions 已启用
- 仓库权限允许 workflow 写 `contents`
- 默认的 `GITHUB_TOKEN` 没有被仓库策略禁用 Release 写权限

当前 workflow 已声明：

```yaml
permissions:
  contents: write
```

### 2. 自动发布触发规则

工作流支持两种触发：

1. `push tags: v*`
2. `workflow_dispatch`

其中：

- **标签触发**：真正发布到 GitHub Releases
- **手动触发**：只构建并上传 Actions artifact，不发 Release

### 3. 正式自动发布步骤

#### 第一步：修改版本号

先修改项目根 [package.json](../package.json) 中的：

```json
"version": "0.1.0"
```

例如改成：

```json
"version": "0.1.1"
```

#### 第二步：提交代码

```powershell
git add .
git commit -m "release: v0.1.1"
```

#### 第三步：创建标签

标签名必须和 `package.json` 版本一致，并且带 `v` 前缀：

```powershell
git tag v0.1.1
```

#### 第四步：推送分支和标签

```powershell
git push origin main
git push origin v0.1.1
```

或者一次性：

```powershell
git push origin main --follow-tags
```

### 4. 工作流内部会做什么

标签推送后，GitHub Actions 会自动：

1. checkout 代码
2. 安装 Node.js 22
3. 校验 `package.json.version` 与标签版本一致
4. 执行 `npm ci`
5. 执行 `npm run build`
6. 执行：

```powershell
npx electron-builder --config electron-builder.config.cjs --win nsis --publish always
```

7. 自动发布到当前 GitHub 仓库的 Releases

### 5. 版本校验规则

工作流内置严格校验：

- 标签 `v0.1.1`
- `package.json` 版本 `0.1.1`

这两者必须完全一致。

如果不一致，工作流会直接失败，例如：

- 标签是 `v0.1.1`
- 但 `package.json` 还是 `0.1.0`

则不会生成 Release。

### 6. 手动触发 workflow_dispatch 的用途

你也可以在 GitHub Actions 页面手动运行这个 workflow。

手动触发时：

- 会构建安装包
- 会上传 Actions artifact
- **不会发 GitHub Release**

适合以下场景：

- 先验证云端打包是否正常
- 检查图标、签名、依赖、构建环境问题
- 暂时不想对外发布版本

## 四、自动更新与发布的关系

已安装客户端自动更新依赖的是：

- 新版本安装包
- 更新元数据
- 它们都已经发布到 GitHub Releases

所以正确链路是：

1. 生成新版本安装包
2. 发布到 GitHub Releases
3. 旧客户端自动检测到新版本
4. 客户端后台下载并提示安装

如果你只是：

- 本地打了一个 exe
- 但没有发布到 GitHub Releases

那么旧客户端**不会**收到更新。

## 五、常见问题

### 1. 只 push 了 commit，没有自动发版

原因：

- 当前 workflow 只监听 `v*` 标签，不监听普通分支提交

解决：

- 补推标签，例如 `git tag v0.1.1 && git push origin v0.1.1`

### 2. 标签推了，但 workflow 失败

先检查：

- `package.json` 中的 `version`
- 标签名是否为 `vX.Y.Z`

最常见错误是：

- `package.json` 是 `0.1.1`
- 标签打成了 `v0.1.2`

### 3. 本地打包时报 `ETIMEDOUT`

这是网络问题，不是业务代码问题。通常发生在 `electron-builder` 下载 Electron / NSIS / 相关构建资源时。

建议：

```powershell
$env:LOCALAPPDATA='E:\idea_space\blog_tool\.cache\localappdata'
$env:APPDATA='E:\idea_space\blog_tool\.cache\appdata'
$env:ELECTRON_CACHE='E:\idea_space\blog_tool\.cache\electron-cache'
$env:ELECTRON_BUILDER_CACHE='E:\idea_space\blog_tool\.cache\electron-builder-cache'
```

然后重试打包。

### 4. 本地执行 `dist:win:publish` 但没有上传成功

检查：

- `GH_TOKEN` 是否设置
- `GITHUB_REPOSITORY` 是否设置为 `owner/repo`
- Token 是否有当前仓库写 Release 的权限

### 5. 新版本已经发布，但客户端没提示更新

检查：

- 当前运行的是不是**安装版**，而不是开发态
- 新版本是否真的到了 GitHub Releases
- Release 里是否包含更新元数据文件
- 当前安装包是否来自支持自动更新的打包版

## 六、推荐操作规范

推荐日常发布流程：

1. 本地改代码
2. 更新 `package.json.version`
3. 本地执行 `npm run dist:win` 验证安装包
4. 提交代码
5. 打 `vX.Y.Z` 标签
6. 推送分支和标签
7. 让 GitHub Actions 自动发布正式 Release

如果是紧急修复，也可以：

1. 本地验证通过
2. 设置 `GH_TOKEN` 与 `GITHUB_REPOSITORY`
3. 执行 `npm run dist:win:publish`

但长期推荐仍以 **GitHub Actions 标签发布** 为主，因为流程更稳定、可追溯。
