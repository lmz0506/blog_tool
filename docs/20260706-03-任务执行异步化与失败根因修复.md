# 20260706-03 任务执行异步化与失败根因修复

## 变更概述

任务执行改为异步后台模式（前端轮询状态），修复"同一任务失败多次才成功"的根因（dev 模式下 server 自动重启杀死执行中的 Agent），修正时间显示时区、发布误提交无关文件等问题；配置中心按需求调整。

## 一、任务 27 反复失败的排查结论

`task_runs` 中 task_id=27 的 4 条记录：#12/#13/#14 失败、#15 成功（349 秒）。三次失败记录的 stdout/stderr/result 全为空，`finished_at` 是"启动僵尸恢复"逻辑盖的章 —— **失败原因不是 Agent 出错，而是 server 进程在任务执行中途重启**，重启把执行中的 codex 子进程杀掉，启动恢复逻辑将运行记录标记为 failed。

重启来源：`server/package.json` 的 dev 脚本是 `node --watch src/index.js`，watch 模式下**进程异常退出会自动重启、源码文件变动也会重启**；叠加旧的同步阻塞设计（执行时整个界面卡住数分钟），用户以为卡死而重启服务，形成"执行→重启→失败→重试"的循环。

**stderr_text 说明**：存放的是 Agent（codex）的过程日志——思考过程、工具调用、写文件的 diff 流水，成功运行有 139KB 属正常现象；最终结果 JSON 在 stdout_text。stderr 大≠出错。

**执行慢说明**：写一篇完整文章 Agent 需要 5-6 分钟，属正常耗时。

## 二、修复与改造

### 1. 任务执行异步化

| 文件 | 方法 | 变更 |
|---|---|---|
| `server/src/services/taskRunnerService.js` | `beginTaskExecution` / `completeTaskExecution`（拆分自 executeTask） | 校验+置 running 与实际执行拆开；新增进程内执行锁 `executingTasks`（Set），同一任务并发触发返回 409「该任务正在执行中，请勿重复执行」 |
| 同上 | `executeTaskInBackground`（新增） | 同步完成校验并置 running 后立即返回任务，Agent 在后台执行，完成/失败写库并打日志；`executeTask`（同步等待版）保留给定时调度器用 |
| `server/src/routes/tasks.js` | `POST /:taskId/run` | 改为调用 `executeTaskInBackground`，返回 202 `{ task, async: true }`，不再阻塞 HTTP 请求最长 15 分钟 |
| `web/src/App.jsx` | 轮询 effect | 存在 running 状态的任务/草稿时每 5 秒静默刷新 tasks/runs/drafts（不触发全局 loading） |
| 同上 | `handleRunTask` / `renderTaskDetail` | 提交后提示"已开始后台执行"并跳详情页；详情页 running 状态显示旋转图标提示条（可离开页面） |

### 2. 防止 server 重启杀任务

| 文件 | 变更 |
|---|---|
| `scripts/dev.mjs` | `npm run dev` 拉起的 server 改用 `npm run start`（无 --watch）；需要热重载调试时单独用 `npm run dev:server` |
| `server/src/index.js` | 新增 `unhandledRejection` / `uncaughtException` 进程级兜底：只记录日志不崩溃 |
| `server/src/services/executorService.js` | `writePromptToStdin`：给子进程 stdin 挂 error 监听，子进程提前退出时避免 EPIPE 未捕获异常导致进程崩溃 |

### 3. 发布误提交修复

**问题**：task 27 的发布提交里混入了博客仓库中本就存在的未提交文件（assets/css/tailwind.css 等）。

| 文件 | 方法 | 变更 |
|---|---|---|
| `server/src/services/gitPublishService.js` | `collectAssetFiles` | 连带提交范围从 `assets/` 收窄到 `assets/images/`（文章配图目录），不再席卷仓库其他改动 |

### 4. 时间时区修复

**问题**：SQLite `CURRENT_TIMESTAMP` 写入的是 UTC 时间但不带时区标记，前端 `new Date()` 按本地时间解析导致显示偏差 8 小时。

| 文件 | 方法 | 变更 |
|---|---|---|
| `web/src/App.jsx` | `toDateValue`（新增）、`formatDateTime` / `formatFullDate` | `YYYY-MM-DD HH:MM:SS` 格式补 `Z` 按 UTC 解析；格式化统一指定 `timeZone: "Asia/Shanghai"` |

### 5. 配置中心调整

| 变更 | 位置 |
|---|---|
| 「扫描仓库分类」按钮从「仓库与发布」移到「分类管理」的分类列表面板 | `web/src/App.jsx` `renderSettingsRepository` / `renderSettingsCategories` |
| 扫描发现的新分类默认加入默认任务池 | `server/src/services/repositoryService.js` `scanRepository`（INSERT is_default_pool=1） |
| 新增自定义分类表单默认勾选"加入默认任务池" | `web/src/App.jsx` `initialCategoryForm` |
| 存量分类一次性迁移全部加入默认任务池（settings 表 `migration_default_pool_all` 标记，之后仍可单独取消） | `server/src/services/storage/database.js` `migrateCategoriesDefaultPool` |

## 三、验证记录

- 全部 server/scripts 文件 `node --check` 通过，`vite build` 通过
- 实测：对已完成任务执行 run 返回 409 中文提示；迁移后 8 个分类全部在默认任务池

## 四、后续可优化（未在本次处理）

1. 草稿生成（draft-plan）仍是同步阻塞（约 30 秒-3 分钟），可按同样模式异步化
2. Windows 下超时杀进程只杀父进程，codex 孙进程可能残留
3. 任务执行无排队机制：不同分类的任务可同时执行多个 Agent，机器负载高时可考虑串行队列
