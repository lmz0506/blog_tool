# 20260703-01 稳定性修复与发布链路补全

## 变更概述

本次变更针对 blog_tool 与 blog（Jekyll 博客）的对接缺口和稳定性问题，完成 1 个功能性 Bug 修复、4 项发布链路补全、2 项健壮性修复、1 项新功能（定时调度）以及死代码清理。

## 一、后端变更

### 1. 任务/草稿卡死 running 状态修复

**问题**：生成草稿和执行任务时先把状态置为 `running` 再启动子进程，一旦执行器抛异常（如执行器 ID 不存在），状态永久停留在 running，且删除接口拒删 running 记录，分类被 `hasRunningDraftForCategory` 永久锁死，重启服务也不恢复。

| 文件 | 方法 | 变更 |
|---|---|---|
| `server/src/routes/drafts.js` | `POST /generate` handler | `runExecutor` 外包 try/catch，异常时 `updateDraftStatus(draftId, "failed")` 并记录错误日志后再抛出 |
| `server/src/services/taskRunnerService.js`（新增） | `executeTask` | 执行器异常时回写任务状态为 `failed` |
| `server/src/services/maintenanceService.js`（新增） | `recoverStaleRunningState` | 服务启动时把所有残留 running 状态的 plan_drafts / tasks / task_runs 记录标记为 failed |
| `server/src/index.js` | 启动流程 | 初始化数据库后调用 `recoverStaleRunningState`，有恢复记录时打 warn 日志 |

另外，失败任务现在允许重新执行：`executeTask` 的状态检查从"仅 pending"放宽为"pending 或 failed"。

### 2. 任务执行逻辑重构为独立服务

**变更**：原 `routes/tasks.js` 中 `POST /:taskId/run` 内联的执行流程（约 70 行）抽取为 `server/src/services/taskRunnerService.js` 的 `executeTask(taskId, { executorId })`，供路由和定时调度器共用。冲突类错误（非 pending/failed、分类占用）带 `statusCode = 409`，路由据此返回正确状态码。

### 3. 发布前校验 front-matter 与文章路径（新增）

**问题**：原流程盲信 Agent 报告的 articlePath，直接 git add/commit，不校验文件是否存在、是否在文档目录内（存在路径穿越风险）；front-matter 完全依赖 Agent 自觉，缺字段会导致博客渲染异常。

| 文件 | 方法 | 说明 |
|---|---|---|
| `server/src/services/taskRunnerService.js` | `resolveArticleFile(repository, articlePath)` | 解析绝对路径，校验必须位于 `{仓库}/{docsDir}` 内（防穿越），文件必须真实存在 |
| 同上 | `validateArticleFrontMatter({...})` | 用 gray-matter 解析文章，自动补齐缺失字段：`layout: doc`、`title`（回退到 Agent 报告标题/文件名）、`category`（用分类展示名）、`date`（当天，`'YYYY-MM-DD'` 格式）、`tags`（默认 `[展示分类名]`），有修补时回写文件 |

校验失败（路径穿越/文件不存在）任务标记 failed，不进入 git 发布。补齐的字段记录在发布结果 `fixedFrontMatterFields` 中。

### 4. 分类目录名与展示名分离

**问题**：blog 中目录名和文章 front-matter 的 category 值可以不同（`devops/` 目录 → `DevOps` 分类、`frontend/` → `前端开发`），原实现把目录名当分类名，新文章会分到错误分组。

| 文件 | 方法 | 变更 |
|---|---|---|
| `server/src/services/storage/database.js` | `migrateSchema` | categories 表新增 `display_name TEXT` 列 |
| `server/src/services/repositoryService.js` | `readArticleMeta` | 额外返回文章 front-matter 中的 `category` 值 |
| 同上 | `pickDisplayName`（新增） | 取该目录下文章中出现最多的 category 值作为展示分类名 |
| 同上 | `scanRepository` | 扫描时把展示名写入 `categories.display_name` |
| `server/src/services/categoryService.js` | `listCategories` / `getCategoryById` | 返回值增加 `displayName`（无则回退目录名） |
| `server/src/services/taskService.js` | `buildTaskPrompt` / `createTaskPromptContext` | 写作 Prompt 中：保存目录用目录名，front-matter 的 category 用展示名，并显式给出完整 front-matter 模板要求 |

### 5. git 发布支持多文件提交（含图片）

**问题**：原来只 `git add` 单个 md 文件，Agent 生成的配图（`assets/images/docs/`）会漏提交。

| 文件 | 方法 | 变更 |
|---|---|---|
| `server/src/services/gitPublishService.js` | `collectAssetFiles`（新增） | 从 `git status` 中收集 `assets/` 下新增/修改的文件 |
| 同上 | `publishArticleToGit` | 文章 + 资源文件一起 add 和 commit，返回值新增 `files` 字段列出提交的所有文件 |

### 6. 发布结果持久化

| 文件 | 方法 | 变更 |
|---|---|---|
| `server/src/services/storage/database.js` | `migrateSchema` | tasks 表新增 `publish_result TEXT` 列 |
| `server/src/services/taskService.js` | `saveTaskPublishResult`（新增） | 保存发布结果 JSON（committed/pushed/branch/files/fixedFrontMatterFields/error） |
| 同上 | `listTasks` | 返回值增加 `publishResult`（`parsePublishResult` 解析） |

git 发布抛异常不再让任务卡在半途：任务仍标记 done，错误写入 `publish_result.error`，可在任务详情中看到。

### 7. 定时调度（新功能）

**问题**：`scheduledDate` 只存不调度，到期任务不会自动执行，自动化闭环缺失。

| 文件 | 方法 | 说明 |
|---|---|---|
| `server/src/services/schedulerService.js`（新增） | `startScheduler(logger)` / `tick` | 每 60 秒检查一次：开关开启时，取执行日期已到的最早 pending 任务，用默认执行器调用 `executeTask` 自动执行；同一时刻只跑一个任务 |
| `server/src/services/settingsService.js` | `getDefaultPlanSettings` / `updateDefaultPlanSettings` | 新增 `autoScheduleEnabled` 设置（默认关闭） |
| `server/src/services/storage/database.js` | `seedSettings` | 种子设置新增 `auto_schedule_enabled = "false"` |
| `server/src/index.js` | 启动流程 | `app.listen` 回调中调用 `startScheduler(logger)` |

### 8. 草稿批量确认端点（新增）+ 重复建任务修复

**问题**：前端"批量加入"按钮实际调用整批确认接口，勾选被忽略；且原 `confirmDraft` 会把已单独确认过的任务项再次建成任务（重复）。

| 文件 | 方法 | 变更 |
|---|---|---|
| `server/src/services/draftService.js` | `confirmDraft(draftId, itemIds = null)` | 支持只确认指定 ID 子集；始终排除已 confirmed 的任务项；所有任务项确认完毕才把批次标记 confirmed |
| `server/src/routes/drafts.js` | `POST /:draftId/confirm-items`（新增） | 接收 `{ itemIds: [] }`，空数组返回 400 |

### 9. 执行器更新校验

| 文件 | 方法 | 变更 |
|---|---|---|
| `server/src/services/executorService.js` | `updateExecutor` | `argsTemplate` 必须是字符串数组（字符串输入自动 JSON.parse，失败报错），`timeoutMs` 必须 ≥ 1000，name/command 去空白且不允许置空 |

### 10. 清理

- 删除 `routes/drafts.js` 中与 `/api/repository` 重复的 `GET /drafts/repository/context` 端点。
- 删除 Prompt 上下文中无效的 `resultFile` 残留参数（`drafts.js`、`tasks.js`）。
- 删除 `server/src/config.js` 中无人引用的 `logsRoot` 导出。

## 五、遗留事项（未在本次处理）

1. 生成/执行接口仍为同步长阻塞（最长 900 秒），建议后续改为"任务队列 + 前端轮询"。
2. 执行器命令等绝对路径仍写死在种子数据中（可在设置页修改，但换机器首次启动需手动调整）。
3. Windows 下超时杀进程只杀父进程，`.cmd` 启动的子进程可能残留。
4. `web/dist` 构建产物尚无 server 静态托管，生产部署仍需 Vite dev server。
5. blog_tool 项目 `.git` 为空目录，无版本控制历史，建议初始化仓库。
