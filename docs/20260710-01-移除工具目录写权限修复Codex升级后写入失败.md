# 20260710-01 移除工具目录写权限，修复 Codex 升级后写入失败

## 问题现象

桌面版升级到 0.1.2 后（2026-07-10 13:56 完成升级），13:59 的写作任务（run #23）失败：Codex 无法写入任何文件——连博客仓库根目录的探测文件都写不进去，最终 Agent 如实返回 `status: "blocked"`（"Local filesystem write tools failed"）。stderr 中每次命令执行都报同一个错误：

```
ERROR codex_core::exec: exec error: windows sandbox: helper_unknown_error: setup refresh had errors
```

对照组：升级前一天（2026-07-09 11:46，run #22）同样的配置、同一版 codex（0.143.0，7月9日 10:13 更新）执行成功，文章正常写入仓库。

## 根因分析

Codex 0.143 在 Windows 上启用了提权沙箱（`~/.codex/config.toml` 中 `[windows] sandbox = "elevated"`）：

1. Codex 不用当前用户账号执行命令，而是用专用受限账户（本地组 `CodexSandboxUsers` 的成员 `CodexSandboxOffline` / `CodexSandboxOnline`）运行，实现操作系统层面的写入边界。
2. 因此每次启动时，沙箱要执行 "setup refresh"：把受限账户的写授权（ACL 条目）登记到**每一个**可写根目录上。**任何一个目录登记失败，整个沙箱初始化失败，所有写入功能瘫痪**——不是只跳过那一个目录。
3. 本工具调用 codex 时传了两个可写目录：博客仓库 `{runDirectory}`（属主为当前用户，授权早已登记，一直正常）和工具安装目录 `{toolRoot}`（`F:\BlogTool\resources\app-runtime`）。
4. 升级 = NSIS 卸载重装，`F:\BlogTool` 整个删除重建：旧目录上已登记的沙箱授权随之丢失，且新目录属主为 **Administrators**（安装器提权写入），普通用户无权修改其 ACL；由本工具后台静默拉起的 codex 又没有弹 UAC 提权的通道 → 授权登记失败 → 沙箱初始化整体失败 → 博客仓库被"连坐"。
5. `--add-dir {toolRoot}` 是历史遗留参数（早期 Prompt/结果走工具目录下的文件）。现在 Prompt 走 stdin（`-`）、结果走 stdout，写作 Prompt 还明确禁止写结果文件，Agent 完全不需要工具目录的写权限。

结论：**每次升级都会复发**，根治办法是不再把安装目录交给 codex 沙箱。

## 修复内容

### 后端

| 文件 | 方法 | 变更 |
|---|---|---|
| `server/src/services/storage/database.js` | `seedExecutors` | codex 种子参数模板移除 `"--add-dir","{toolRoot}"`，变为 `["exec","-s","workspace-write","--add-dir","{runDirectory}","-"]` |
| `server/src/services/storage/database.js` | 新增 `migrateCodexExecutorRemoveToolRootAccess` | 幂等迁移：扫描所有 codex 类型执行器，从参数模板中移除成对的 `"--add-dir","{toolRoot}"`；在 `migrateSchema` 中于 `migrateCodexExecutorRunDirectoryAccess` 之后调用，保证旧格式（只有 toolRoot、缺 runDirectory）先补齐 `{runDirectory}` 再删 `{toolRoot}` |

`executorService.js` 的模板上下文仍保留 `toolRoot` 键，用户自定义模板中的 `{toolRoot}` 占位符仍可解析。

### 前端

无改动（参数模板不暴露在界面上）。

### 已安装环境的即时修补

新版本发布前，已直接对本机已安装应用的数据库 `F:\BlogToolData\storage\blog-tool.db` 执行了与迁移等价的 UPDATE（移除 codex 执行器参数中的 `--add-dir {toolRoot}`）。执行器配置每次运行时从数据库现读，无需重启应用即生效。

## 验证

对迁移逻辑做了三组测试（在临时目录中运行 `initializeDatabase`）：

1. **已安装应用数据库副本**：`{toolRoot}` 对被移除，`{runDirectory}`、工作目录、claude 执行器均不受影响，业务数据完整（12 个任务）
2. **全新数据库种子**：直接生成不含 `{toolRoot}` 的参数模板
3. **历史旧格式**（`["exec","-s","workspace-write","--add-dir","{toolRoot}","-"]`）：两个迁移正确组合，结果为 `["exec","-s","workspace-write","--add-dir","{runDirectory}","-"]`

端到端验证方式：在界面上重新执行升级后失败的任务，确认文章能正常写入 `E:\idea_space\blog\_docs\` 并发布。

## 附注

- 升级时 `F:\BlogTool\storage` 旧存储目录被删除属另一问题，已由 0.1.2 的存储迁移功能解决（存储移至 `F:\BlogToolData\storage`）；本次确认旧目录已不存在，升级前的历史数据无法恢复。
- 观察到 run #23 虽然 Agent 返回 `status: "blocked"`，但因带有 `BLOG_TOOL_TASK_DONE` 标记被记为 success（`executorService.runExecutor` 的判定只看标记）；任务本身由后续文章校验兜底判失败，行为正确但运行记录状态有误导性，待后续优化。
